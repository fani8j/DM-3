# Human Activity Recognition (HAR) Pipeline

**NYCU Data Mining Spring 2026 — Assignment 3**

This repository contains the code for Assignment 3, a Kaggle Human Activity Recognition competition. The task is to classify wrist-worn accelerometer windows (300 rows x 6 channels) into six activity labels `{0, 1, 2, 3, 4, 5}` and produce Kaggle-compatible submission CSV files.

The raw Kaggle data, cached model probabilities, checkpoints, generated submissions, report PDFs, and figures are not intended to be committed to GitHub. They are downloaded or regenerated locally. The report focuses on experiment design, data analysis, feature engineering, ablation studies, and leaderboard interpretation. This README focuses on how to run the code, what the repository contains, and which submission files correspond to the reported Kaggle scores.

## Requirements

- Python >=3.10,<3.13
- [uv](https://docs.astral.sh/uv/) package manager

## Installation

```bash
uv sync
```

This resolves all dependencies from the pinned `uv.lock` file, ensuring identical package versions across machines.

If using the already-created local virtual environment:

```bash
source .venv/bin/activate
```

## Dataset Download

Download the competition files with `kagglehub`:

```bash
uv run --with kagglehub python scripts/download_data.py
```

This calls:

```python
import kagglehub

path = kagglehub.competition_download("nycu-data-mining-assignment-3")
print("Path to competition files:", path)
```

By default, `scripts/download_data.py` creates local symlinks for:

```text
train/
test/
sample_submission.csv
```

The default config expects:

```text
train/train
test/test
sample_submission.csv
```

If the files are already present locally, the script leaves them unchanged.

## Data Validation

Verify dataset integrity before training (checks row counts, column presence, label ranges, user-set disjointness, and File_Id ↔ Sample_Submission alignment):

```bash
uv run har-validate-data --config configs/default.toml
```

## Training

Train with all configured validation strategies (group K-fold and random split):

```bash
uv run har-train --config configs/default.toml
```

Train with a specific validation strategy:

```bash
uv run har-train --config configs/default.toml --override validation.strategies='["group_kfold"]'
uv run har-train --config configs/default.toml --override validation.strategies='["random_split"]'
```

The top-level experimental scripts used for the final blend are:

```bash
.venv/bin/python train_ensemble.py
.venv/bin/python train_tcn_oof.py
.venv/bin/python train_deep_v2.py
.venv/bin/python train_ensemble_v3.py
.venv/bin/python train_final.py
```

These scripts write model probabilities and feature matrices to `feature_cache/`, then `make_cached_blend.py` combines them into final submissions.

## Prediction

Generate a Kaggle submission from a trained checkpoint:

```bash
uv run har-predict --config configs/default.toml --checkpoints checkpoints/<run_id>/best.pt --output submissions/submission.csv
```

Replace `<run_id>` with the UUID of the training run (printed during training and recorded in `experiments/runs.jsonl`).

## Ablation Aggregation

Aggregate all experiment records into ablation tables (preprocessing, features, architecture, splitter):

```bash
uv run har-aggregate --experiments-dir experiments --out experiments/
```

## End-to-End

Train and predict in one command (validate → train → predict):

```bash
uv run har-run --config configs/default.toml --output submissions/final.csv
```

## Reproducibility

This pipeline is designed for exact reproducibility:

- **Seeds**: A single integer seed controls Python `random`, NumPy, and PyTorch (CPU + CUDA) random number generators. Set via `[reproducibility] seed` in the config.
- **Deterministic flags**: When `reproducibility_mode = true`, PyTorch deterministic algorithms are enabled and cuDNN benchmark mode is disabled (`torch.use_deterministic_algorithms(True)`, `cudnn.benchmark = False`).
- **`uv.lock` policy**: The lock file pins all dependencies (including transitive) to exact versions. Always commit `uv.lock` and run `uv sync` (not `uv pip install`) to guarantee identical environments.
- **Manifest alongside checkpoints**: Every saved checkpoint includes a `manifest.json` recording the full config, seed, Git commit hash, preprocessor statistics, and feature configuration. This ensures predictions can be reproduced from any checkpoint without relying on external state.

Two consecutive `har-run` invocations with the same config, data, and lock file produce submission files with identical SHA-256 digests.
